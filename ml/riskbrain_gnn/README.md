# RiskBrain GNN

This folder contains the first GPU-side graph model for iBank RiskBrain.

The current model is an edge-classification GraphSAGE baseline implemented with plain PyTorch. It intentionally avoids PyTorch Geometric extension dependencies so that the first rented-GPU training loop is stable.

## Input Files

Export a graph batch from iBank with `RiskGraphExporter`, then copy the export directory to the GPU server.

Expected files:

- `nodes.csv`
- `edges.csv`
- `labels.csv`
- `graph_meta.json`
- `edge_features_v2.csv` (optional, generated by the current exporter)
- `feedback_labels.csv` (optional, generated when manual review feedback exists)

## IBM Multi-GNN Paper-Protocol Track

The sampled iBank track above is useful for system integration, but it is not a paper-protocol reproduction. The paper
track uses the full `HI-Small_Trans.csv` file and the official Multi-GNN temporal split logic:

- Raw full HI-Small rows: `5,078,345`.
- Positive labels: `5,177`.
- Positive rate: `0.1019%`.
- Official day-boundary split produced on this file: train `63.98%`, validation `19.01%`, test `17.01%`.
- Split days: train days `0-5`, validation days `6-7`, test days `8-17`.
- Primary metric: minority-class F1, selected by best validation F1.
- Official graph history protocol: train graph uses train edges, validation graph uses train+validation edges, test graph
  uses all edges.

## Notes

- Task: transaction-edge fraud/laundering classification.
- Positive label: `label_fraud=1`.
- Legacy split: stratified train/validation/test over edges.
- Leakage-safe split: chronological train/validation/test over edges, preferably generated once with
  `build_temporal_splits.py` and reused through `--split-file`.
- V1 node features: graph aggregate features, historical fraud-degree features, and node type one-hot.
- V2 node features: graph aggregate features without fraud-degree leakage, plus node type one-hot.
- V1 edge features: amount transform, edge type one-hot, currency one-hot, source-step feature.
- V2 edge features: V1 edge features plus endpoint degree/amount context, batch-local node activity,
  amount-ratio features, time-of-day/weekend/night indicators, and manual review feedback weights.
- Model: two-layer mean GraphSAGE node encoder plus MLP edge classifier.
- V3 architecture: two-layer directed GraphSAGE node encoder with separate inbound/outbound aggregation,
  edge-attribute encoder, edge-state update MLP, and optional hard negative mining over high-scored normal edges.
- V4 leakage-safe mode: historical edge features only, feedback off, time split, train-only normalization,
  and train-only adjacency for evaluation. This is the preferred backtest setup when checking for leakage.
- V5 temporal feature baseline: current transaction features plus prefix-only historical graph statistics, trained on
  the same fixed temporal split as V4. It is the interpretability baseline for deciding whether GNN adds value.
- V6 temporal-feature GNN: directed-edge-sage on the exact V5 temporal feature matrix. This tests whether graph message
  passing adds lift after strong prefix-only features are already present.
- V7 line-graph feature baseline: V5 temporal features plus transaction-chain features derived only from prior
  transactions. This tests whether explicit edge-to-edge path structure adds lift.
- V8 line-graph neural model: transaction edges become line-graph nodes, with temporal-past edge-to-edge message
  passing over related prior transactions. Current V8 is useful as a graph-research branch and high-confidence graph
  evidence source, but it does not yet replace the V5/V7 tree baselines as the primary operating model.
- V9 relation-aware line-graph GNN: V8 plus relation-specific temporal message channels and learnable relation gates.
  It improves over V8 on AP, F1, and missed positive count, making it the current neural graph branch to keep, but V5/V7
  remain stronger production baselines until a calibrated ensemble or stronger graph architecture proves lift.
- IBM Multi-GNN reproduction: official PyG code now runs on the iBank fixed temporal split through the converted
  `formatted_transactions.csv` data path. Corrected 100-epoch runs show Multi-GIN is much stronger than the initial
  smoke test suggested, but still below V9 under hard-label F1. The next meaningful work is a stricter reproduction
  harness with probability export, AP/oracle-F1, and imbalance-aware training.
- A model run is not considered a valid leakage-safe backtest unless `leakage_guard.py` returns no FAIL checks.
